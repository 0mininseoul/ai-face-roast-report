## 1. 피봇 비전 (사용자 원문)

- 현재 서비스: AI 얼평 (풍자 톤, 데스크톱+웹캠, MediaPipe + Gemini + Supabase)
- 피봇 후: **AI 관상 기반 독서 추천 서비스**
- B2B 목표: **대학 중앙도서관 입점 계약**
- 핵심 플로우: 학생이 이름/학번/선호 카테고리/성별/생년월일 입력 → 실시간 웹캠 얼굴 vision 분석 → 관상 + 가벼운 사주 기반 해석 → 도움이 될 책 3권 추천 (청구기호 + 도서관 내 위치 안내)
- MVP 전제: 학교 도서관 책 DB 없음, 학생 DB 없음
- 톤 워딩: "AI 관상 독서 처방" 컨셉 살림 (재미 요소 + 학습 가치)

## 2. 확정된 의사결정 (이번 세션)

| 항목 | 결정 |
|---|---|
| **배포 형태** | 도서관 한 켠 **부스형 일반 윈도우 PC + 웹캠** (키오스크 X). 학생 walk-up. 야외 행사도 동일 셋업으로 운영 가능. |
| **데이터 저장** | Supabase에 유저 세션/결과 DB화 필수 |
| **입력 필드** | 이름, 학번, 좋아하는 독서 카테고리, 성별(직접 선택), **생년월일(시간 X)** |
| **분석 방식** | **관상 + 간단 사주(년주/월주 수준)** — 태어난 시간은 받지 않음. 입력 마찰 최소화 + "사주" 워딩으로 마케팅 후크 확보 |
| **책 데이터 소스** | **국립중앙도서관(NLK) 등 공공 데이터셋 임포트** (실계약 시 학교 OPAC로 갈아끼울 수 있는 어댑터 구조 필수) |
| **추천 후 핸드오프** | **청구기호 + "도서관 어디에 있는지" 안내**. MVP에선 더미 데이터로 시뮬레이션, 실계약 시 OPAC 연동. |
| **추천 권 수** | 3권 |
| **얼굴 이미지 보존** | **저장하되 7일 후 자동 삭제** (Supabase cron/storage expiration). 입력 폼에서 별도 명시적 동의(민감정보 동의) 필수. |

## 3. 아직 결정 안 된 / 다음 세션에서 짚어야 할 질문 (순서대로)

다음 모델은 한 번에 하나씩 객관식으로 물어보세요. 우선순위 순:

1. ~~**프라이버시 / 데이터 보존 정책**~~ — **결정됨: 7일 후 자동 삭제.** 추가로 확정 필요한 세부:
   - 학번/이름 해싱? (DB 평문 vs 해시) — 권장: 평문 보관하되 어드민은 학번 끝자리 마스킹
   - 학생 본인이 사전 삭제 요청 가능한 인터페이스 (학번 + 생년월일 lookup)
   - 도서관 어드민이 얼굴 이미지를 볼 수 있는지 (권장: 못 봄, 메트릭/결과만)
   - Supabase Storage bucket 분리 (faces/ 7일 TTL, results/ 7일 TTL or 별도)
2. **추천 로직 아키텍처** — 다음 3가지 중:
   - (A) 얼굴 메트릭 → LLM 자유 해석 + 책 시드 안에서 매칭 (LLM 한 번 호출)
   - (B) 얼굴 메트릭 → 사전 정의 "덕목/성향" 태그 추출 → 책 태그와 룰 기반 매칭
   - (C) 하이브리드 (LLM이 태그 추출 → 룰로 필터 → LLM이 최종 카피)
   - 추천: B2B 신뢰성 관점에서 (C) 하이브리드가 환각 통제 + 다양성 확보 균형
3. **결과물 톤 / 카피 방향**
   - 진중한 '독서 처방전' (학습 도구)
   - 가벼운 '오늘의 책' (재미 + 입소문)
   - 둘 다 하이브리드
4. **세션 종료 후 데이터 처리**
   - 자동 삭제(예: 24시간) vs 영구 저장
   - 학생이 결과 다시 보고 싶을 때 학번으로 재접근 가능?
5. **B2B 가치 제안 (영업 피치)**
   - 도서관 KPI는? (대출 권수, 방문자 수, 신간 노출 등)
   - 입점 비용 모델 (무료 입점/매출 분배/구독 라이선스)
6. **행사 모드 vs 상시 모드 분리 여부** — 한 빌드로 둘 다 vs 별도 모드
7. **다국어** — 한국어만 시작? 교환학생용 영어 페이지?
8. **세션 길이 / 처리 시간 목표** — walk-up이라 1~2분 안에 끝나야 함. Gemini 호출 비용/지연 관리 방안

## 4. 기존 코드베이스 (재활용/폐기 자산)

스택: Next.js 14, TypeScript, Tailwind, Framer Motion, MediaPipe Tasks Vision, @google/genai (Gemini), Supabase, pnpm.

**그대로 살릴 자산:**
- 웹캠 캡처 파이프라인 (`src/lib/camera`, `src/lib/capture`, `src/lib/facemesh`)
- MediaPipe FaceLandmarker 메트릭 추출
- Gemini API 호출 구조 (`src/lib/gemini`, `src/lib/analysis`) — 프롬프트만 교체
- Supabase 클라이언트/스키마 (`src/lib/supabase`, `supabase/migrations/`)
- API: `/api/analyze` (재활용), `/api/live-comment` (톤만 변경 또는 제거 가능)
- 어드민 대시보드 (`src/app/admindata/`) — 책 큐레이션/세션 모니터링용으로 재활용
- i18n 인프라 (`src/lib/i18n`, `src/app/[locale]/`) — 영어 추가 시 그대로
- Pretendard 폰트, 로고, 데스크톱 전용 가드

**폐기 또는 재작업:**
- "얼평/풍자" 톤 카피 (`src/lib/copy/`) — 전면 재작성
- 결과 페이지 카피/디자인 (`src/components/result/`, `src/app/result/`) — 책 추천 UI로 재구성
- 카카오톡 공유 (`src/lib/kakao`, `src/app/api/share-*`) — 부스 환경에선 불필요(QR 도입 시 재활용 가능)
- mobile-only 가드 (부스라 데스크톱만, 그대로 두면 됨)
- 효과음 (`src/lib/sound`, `public/sfx/`) — 부스 환경 톤에 맞게 재선정
- `prompts/analyze-system.md`, `prompts/live-comment.md` — 전면 재작성

**새로 만들 것:**
- 입력 폼: 이름/학번/카테고리/성별/생년월일
- 책 시드 데이터 (NLK API 임포트 + LLM 태깅 배치 스크립트)
- 책 데이터 모델 (Supabase 테이블)
- 추천 엔진 (위 3.2의 결정에 따라)
- 결과 페이지 (관상 1단락 + 책 3권 카드 with 청구기호/위치)
- 책 위치 안내 컴포넌트 (MVP는 더미, 어댑터 인터페이스로)
- OPAC 어댑터 인터페이스 (실계약 시 교체 가능 구조)

## 5. 잠정 MVP 스코프 (제안 — 사용자 승인 전)

**Phase 1 (실제 MVP):**
1. 입력 페이지 — 이름/학번/카테고리/성별/생년월일 + 개인정보 동의 체크
2. 웹캠 캡처 + MediaPipe 메트릭 (기존 그대로)
3. Gemini 단일 호출로 구조화 JSON 응답: `{ 관상해석, 사주키워드, 추천도서: [{book_id, 추천이유}, ...3] }`
4. 결과 페이지 — 관상 분석 1단락 + 책 3권 카드 + 청구기호/도서관 위치(더미)
5. Supabase에 세션/결과 저장
6. NLK 공공 데이터에서 카테고리당 ~50권 시드 → LLM 태깅 배치 → 약 300권 책 DB
7. 어드민 페이지에서 책 데이터 검수/수정

**Phase 2 (B2B 데모/입점 영업용):**
- QR로 결과 핸드폰 전송
- 행사 모드 / 상시 모드 분기
- B2B 운영 대시보드 (오늘의 사용자/인기 추천/카테고리 분포)
- 영업 자료/시연 페이지

**Phase 3 (실계약 후):**
- 학교 OPAC API 어댑터 구현 (인터페이스는 Phase 1부터 잡혀 있어야 함)
- 학교 SSO 학번 검증
- 다국어 (영어)
